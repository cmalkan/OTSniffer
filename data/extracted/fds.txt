| Based system with Foxboro MESH network and multiple OS/DCS versions (8.x/9.x). RMP
| DCS System is split into seven areas (WS#1 to WS#7) and is spread across the following
| locations:
| •  RMPCCR
| •  RCC
| •  SIH:
| o  UCR
| o  RCR
| o  NGOD
| o  FAR - 107
| o  FAR - 144
| o  FAR - 153
| The Existing DCS System in RMP has two portions:
| •  Work  group - based  system.
| o  WS#3, WS#4 and RMP Brownfield units in RMPCCR,
| o  UCR,
| o  FAR - 107
| o  FAR - 144
| o  FAR - 153
| •  Security Enhanced  Domain based  System
| o  WS#1, WS#2, WS#5 and WS#7 in RMP  |  CCR/NGOD/RCR
| The existing cybersecurity  infrastructure (Supplied as part of CA/CPC/0297)  in RMP  DCS  consists of
| following components:
| 1.  Cybersecurity servers
| •  Primary Domain Controller (RMPPDC)
| •  Secondary Domain Controller/Centralized ePO (RMPSDC)
| •  Centralized Backup Server (RMPBKP)
| •  WSUS Server (RMWSUS)

| 2.  Redundant ICSS Firewall RMFW01, RMFW02
| 3.  Redundant PIN Root and Backup Root Switches (RMPNRT, RMPNBK)

| RMPN04)

| 8.1  Network Segmentation
| Network segmentation divides the  ICSS n etwork s  into multiple zones  and sub - zones . Each zone has
| a different set of  traffic patterns, security, and  connectivity requirements. Segmentation is  achieved
| by placing firewalls at  zone boundaries . Intrusion prevention functions are embedded in the internal
| segmentation NGFW  devices.
| Firewall filtering strategies:
| •  The base configuration of the Firewall device will be set to deny all communication by default and
| only allow communication by exception to meet a critical application need. This applies to both
| intermittent and interactive user communication, and task - to - task communication between
| devices in these two zones.
| •  Ports and servers frequently used as attack vectors will not be opened through the firewall device.
| •  The fewer the number of ports and services open through the firewall device the better, for OPC
| traffic, FortiGate firewall is an OPC aware device.

| 12  DOMAIN CONTROLLERS
| 12.1  Active Directory Overview
| Microsoft  Windows  Active  Directory  Domain  Services  (ADDS)  provide  centralized  access
| D  |  management of users and resources from a server defined as a Domain Controller.
| Foxboro DCS installs the Active Directory on a Primary Domain Controller during the Control Core
| Services installation. A set of preconfigured user groups and security policies are installed as part of
| the Foxboro ™  Active Directory. These group policies govern the access control and security posture
| of the users and computers that are part of the DCS.
| Domain controllers store information, including security credentials and login information for all
| users’  accounts and resources within the domain in central database. Domain controllers allow a
| central access point for configuring and administering a secure network.
| Foxboro DCS installed with the Enterprise Edition employs the use of Microsoft Active Directory.
| A central and vital component of an Active Directory environment is the Domain Controller (DC).
| Only one Domain Controller can take on the role of the “primary” Domain Controller (PDC) at a
| particular time. As the DC is vital component additional “secondary” Domain Controller (SDC) is
| considered which can take over the primary role if the current pri  |  mary Domain Controller
| experiences a detected failure.
| The new Domain Name to be defined in the form “RMP.local” from which the full computer name
| FQDN must be formed as the following structure: “ KNPC .RMP.local.”
| Note: Domain Name can be changed with the company name preference.
| 12.2  Overview of Domain Controllers Roles
| 12.2.1  Operational Setup
| •  Primary Domain Controller (PDC):
| •  Hosts all FSMO (Flexible Single Master Operations) roles, especially the PDC
| Emulator role.
| •  Responsible for time synchronization, password changes, and acting as the
| preferred DC for clients.
| •  Runs DNS and Global Catalog services for domain - wide authentication.
| •  Secondary Domain Controller (SDC):
| •  Serves as a backup DC and holds a full read/write copy of the AD database.
| •  R uns DNS and Global Catalog services for redundancy.
| •  Participates in the same replication topology as the PDC.
| Both DCs are configured as Global Catalog Servers and DNS Servers for fault tolerance and faster
| logon services.

| 12.2.2  Replication Configuration
| •  Replication Type: Multi - Master Replication
| o  All DCs replicate changes to each other in a multi - master fashion using Active
| Directory Replication Services (ADRS).
| o  Changes to objects (e.g., user accounts, group policies) can be made on any DC and are
| propagated to others.
| •  Replication Schedule:
| o  Intra - Site Replication (within same site): Occurs every 15 seconds by default.
| •  Transport:
| o  Uses RPC over IP for intra - site replication.
| •  Conflict Resolution:
| o  Uses USNs (Update Sequence Numbers) and a last - writer - wins strategy to resolve
| replication conflicts.
| 12.2.3  Failover  Process
| In the event of the Primary DC failure, the following failover process ensures continued domain
| functionality:
| •  Authentication and Authorization:
| o  Clients are automatically redirected to the Secondary DC for login and resource
| access.
| o  The SDC responds to LDAP, Kerberos, and DNS requests without manual intervention.
| •  FSMO Role Failover:
| o  If the Primary DC holding FSMO roles is unavailable:
| o  Short - term:  AD  functions  continue,  but  certain  tasks  (e.g.,  password  lockout
| processing, RID allocation) may degrade.
| o  Long - term: FSMO roles should be seized using PowerShell or NTDSUTIL if recovery is
| not expected.
| •  DNS Failover:
| o  Clients use multiple DNS servers (e.g., IPs of both PDC and SDC) in their NIC settings.
| o  If one DNS server is down, the client automatically queries the other.

| Multiple domains can be grouped into a tree and multiple trees can belong to a forest (the highest
| level).

| OUs act as containers of objects (such as users and workstations); whereas security groups are a
| collection of attributes that can be assigned to an object. Each computer or user can be a member of
| multiple Security Groups.
| NOTE: A user account can be in only one OU but can be a member of any number of security groups.
| 12.3.5  Security Group Policies and Group Policy Objects
| Group policies are applied to OUs, domains, or sites. Group policies provide central administration of
| settings that  can be applied to multiple users and workstations. Group policy determines which users
| have permission to perform certain actions, such as shutting down a workstation or having access to
| the Task Manager.
| Group policies can manage computer  |  - specific settings, user - specific settings, or both. Computer
| policies are applied at boot time. User policies take effect when a user logon occurs. Domain based
| group policies (other than folder redirection and software i nstallation) are not only applied at logon
| time, but also reapply themselves periodically to keep current with any changes.
| Group Policy refresh interval specific for Domain Controllers is every 5 minutes and a different refresh
| interval of 90 minutes is used for all other computers. These refresh intervals are controlled via group
| policy.
| Group policies are processed in a hierarchical order. For example, Local Security Policy is first, then
| site -  level policies, then domain - level policies, and then OU - level policies. Users and workstations that
| are both in a domain and an OU receive setting s both from the domain - level policy (Base Policy) and
| from the OU level policy. If the same policy is configured at multiple levels, the behavior` depends on
| how the policy is configured at the higher level. For example, it is possible to set a “No Overrid e” option
| that prevents a policy from being overridden at a lower level. Otherwise, normally the lower - level
| policy (the last one applied) would win.

| Microsoft Active Directory Certificate Services (AD CS) is a crucial component for establishing a
| secure network infrastructure. It serves as a  foundation for Public Key Infrastructure (PKI), issuing
| and managing digital certificates used for authentication, encryption, and digital signatures. AD CS
| ensures data confidentiality, integrity, and non - repudiation by binding identities to cryptographic
| keys, empowering organizations to securely protect sensitive information and resources.
| The LDAP (Light Weight Directory Access) protocol enables communication with LDAP servers (AD
| servers). Various clients, including Microsoft applications, use the LDAP protocol. By default, LDAP
| communications between client and server applications are not  encrypted. One way to help make
| the LDAP traffic more secure is to tunnel it via SSL/ TLS protocols. A default AD installation does not
| configure LDAP over SSL. Use the information here to configure LDAP over SSL for CCS Enterprise
| Edition installations.
| Following Roles are installed in the Certificate Authority Servers.
| -  Certification Authority.
| -  Certification Authority Web enrollment.
| -  Online Responder.
| -  Certificate Enrollment Policy Web Service.
| 16.1  Public Key Infrastructure (PKI) Architecture
| The PKI is designed as a two - tier hierarchy to balance security and operational manageability. At
| the top is an offline Root CA , which is used solely to sign  subordinate CA certificates and is kept
| powered off and physically secured. Beneath it is an online Issuing CA, which is domain - joined and
| responsible for issuing certificates to users, devices, and services within the control system
| environment. This sepa ration ensures that the Root CA remains isolated from potential threats,
| while the Issuing CA can operate within the network to support certificate lifecycle operations.
| Active Directory Certificate Services. Configuration parameters.
| 16.2  Certificate Authority Configuration
| The Root CA is configured with a shorter certificate lifetime and publishes CRLs to internal - only
| distribution points. It is integrated with Active Directory to support identity - based certificate
| issuance. The CA is hardened by disabling unnecessary roles  and services, and its access is
| restricted to authorized administrators only.
| 16.3  Certificate Revocation Strategy
| Given the air - gapped nature of the environment, Certificate Revocation Lists (CRLs) are the primary
| mechanism for revocation checking. CRLs are published to an internal HTTP server accessible via
| internal DNS. Delta CRLs is enabled to reduce the size and f requency of full CRL downloads. The
| Issuing CA is configured to include CRL Distribution Point (CDP) and Authority Information Access
| (AIA) extensions in all issued certificates, pointing to internal - only URLs.
